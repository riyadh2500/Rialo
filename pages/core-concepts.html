Architecture overview

Rialo is described as a "real-world blockchain" — but technically it's a vertically integrated programmable network. Unlike Ethereum or Solana, which require developers to assemble a stack of external services (oracles, bridges, keepers, relayers), Rialo bakes those capabilities directly into the protocol layer.

The stack can be thought of in five layers:

REX — the execution runtime

The Rialo Extended Execution Runtime (REX) is the heart of the network. It is built on RISC-V, an open instruction set architecture that gives Rialo several advantages over EVM or WASM-based chains:

  • ZK-friendly — RISC-V's simpler instruction set makes it easier to generate zero-knowledge proofs of execution, enabling privacy-preserving computation.
  • Confidential computing — REX integrates with Trusted Execution Environments (TEEs) to run programs in hardware-isolated enclaves, enabling private state and verifiable off-chain computation.
  • Native randomness — True on-chain randomness is a first-class primitive in REX, not a service you bolt on.
  • Performance — RISC-V's clean ISA allows for highly optimized JIT compilation with predictable, stable execution costs.
ℹ️
Why RISC-V over EVM?

The EVM was designed for sequential, isolated contract execution — a model that predates the modern async web. RISC-V gives Rialo a general-purpose foundation that can express async patterns, confidential compute, and hardware-level optimizations without retrofitting workarounds onto an aging design.

VM layers

REX is Rialo's canonical execution environment, but the network supports multiple VM execution surfaces to meet developers where they already are:

⚙️

SVM (Solana VM) Available

Full Solana program compatibility via Sealevel. Existing Rust/Anchor programs can be ported with minimal changes using rialo-sol-lang.

🔷

EVM Planned

Ethereum VM compatibility layer for porting Solidity contracts, enabling access to the large Ethereum developer ecosystem.

🔵

MoveVM Planned

Move language support for developers coming from Sui or Aptos, with resource-oriented programming semantics.

Concurrency model

Parallel execution is a near-universal requirement for high-throughput blockchains. Rialo's concurrency model is designed for correctness first, performance second — avoiding the class of bugs that come from optimistic parallel execution without proper conflict detection.

Transactions in Rialo are organized into sequential batches with consecutive IDs. Each transaction in a batch:

  1. Declares all accounts it needs upfront (no lazy account resolution mid-execution)
  2. Executes against its declared account set
  3. Commits atomically — all state changes land, or none do

This design eliminates the most common source of parallel execution bugs (mid-flight account conflicts) while still enabling high throughput because non-overlapping account sets can be scheduled in parallel across batches.

Event-driven execution

Most blockchains are passive — they wait for transactions to arrive. Rialo is reactive. Programs can register to run in response to:

  • Time triggers — EVERY 60s, AFTER 24h
  • On-chain events — another program emits an event your program is listening for
  • External data — a price feed crosses a threshold, an API returns a specific payload
  • Cross-chain signals — state changes on another chain propagate without a bridge

This is implemented at the protocol level via the Venus DSL, not via off-chain keepers or cron jobs. There is nothing to host, nothing to monitor, nothing to pay a third party for.

rust
rialo! {
    state { last_run: i64 }

    program {
        // Protocol-level cron — no keeper, no bot
        EVERY 1h => {
            let price = price_feed::get("BTC/USD");
            if price < 50_000 {
                SEND alert_subscribers();
            }
        };
    }
}

State model

Rialo programs have two kinds of state:

  • Workflow state — persistent data declared in the state { } block. This survives across async operations and transaction boundaries. The runtime handles serialization, account allocation, and rent automatically.
  • Ephemeral state — local variables within a function call. Discarded after the call completes.

The state model is intentionally simpler than Solana's account model. You don't manage PDAs, account sizes, or rent exemptions manually — the CDK and runtime handle these details, letting you focus on program logic.
